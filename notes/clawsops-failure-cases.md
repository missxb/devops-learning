## 🔥 案例 01：凌晨 3 点的数据库主库宕机

### 故障描述

```
时间：2022-03-15 03:22
现象：MySQL 主库突然宕机，从库无法切换
影响：核心业务全部不可用，持续 45 分钟
```

### 根本原因

1. 主库磁盘满 100%，无法写入 binlog
2. 从库复制延迟 2 小时，切换后数据不一致
3. 没有监控磁盘使用率告警
4. 没有自动故障转移机制

### 时间线

```
03:22 主库磁盘满，写入失败
03:25 监控检测到异常，但告警策略没配置磁盘告警
03:30 用户开始投诉，客服转告运维
03:35 运维接到电话，开始排查
03:40 发现主库磁盘满，尝试清理
03:45 清理失败，决定切换到从库
03:50 发现从库延迟 2 小时，数据不一致，不敢切换
04:00 手动同步数据，强制切换
04:07 业务恢复
```

### 改进措施

| 问题 | 改进 | 负责人 | 状态 |
|------|------|--------|------|
| 磁盘无告警 | 磁盘 80% 告警 | 张三 | ✅ |
| 无自动切换 | 部署 MHA | 张三 | ✅ |
| 从库延迟监控 | 复制延迟 > 60s 告警 | 李四 | ✅ |
| 无应急预案 | 编写故障切换 SOP | 王五 | ✅ |

### 经验教训

> **磁盘监控是最基础也是最重要的监控，别等满了才发现**

## 🔥 案例 02：周五下午的发布灾难

### 故障描述

```
时间：2021-12-24 17:30 (平安夜前)
现象：发布后服务全部启动失败
影响：加班到凌晨 2 点才恢复
```

### 根本原因

1. 周五下午发布，违反发布纪律
2. 新配置格式不兼容旧版本
3. 没有灰度，直接全量发布
4. 没有回滚方案，只能硬修

### 时间线

```
17:30 开始发布 (周五下班前)
17:35 配置更新到配置中心
17:40 服务重启，读取新配置
17:42 服务启动失败，日志报配置解析错误
17:45 发现配置格式不兼容
17:50 尝试回滚配置，但配置中心没有历史版本
18:00 开始手动修改配置，但改了 50 台还有 100 台
19:00 发现改不完，开始写批量脚本
20:00 脚本写好，批量推送配置
21:00 服务逐步恢复
02:00 全部恢复，写故障报告
```

### 改进措施

| 问题 | 改进 | 负责人 | 状态 |
|------|------|--------|------|
| 周五发布 | 禁止周五发布，周一到周四 | 张三 | ✅ |
| 无灰度 | 增加灰度发布流程 | 李四 | ✅ |
| 配置无回滚 | 配置中心增加版本管理 | 王五 | ✅ |
| 无发布检查 | 增加 pre-commit hook | 张三 | ✅ |

### 经验教训

> **周五发布是运维大忌，别让同事陪你过周末**

## 🔥 案例 03：一次误操作删了生产数据库

### 故障描述

```
时间：2020-06-10 14:30
现象：执行 DELETE 语句误删了用户表
影响：10 万用户数据丢失，恢复 4 小时
```

### 根本原因

1. 在生产环境直接执行 SQL
2. SQL 语句写错，少了 WHERE 条件
3. 没有操作审批流程
4. 没有即时备份

### 时间线

```
14:30 开发说需要清理测试数据
14:32 运维直接登录生产库
14:35 执行 DELETE FROM users (忘了加 WHERE)
14:36 发现删多了，立即 Ctrl+C，但已经删了 10 万条
14:40 检查 binlog，准备恢复
15:00 开始从 binlog 恢复
18:30 恢复完成，验证数据
```

### 改进措施

| 问题 | 改进 | 负责人 | 状态 |
|------|------|--------|------|
| 直接操作生产 | 禁止直接操作，必须审批 | 张三 | ✅ |
| 无 WHERE 条件 | DELETE/UPDATE 必须有 WHERE | 李四 | ✅ |
| 无即时备份 | 增加 binlog 实时备份 | 王五 | ✅ |
| 无审计 | 增加 SQL 审计日志 | 张三 | ✅ |

### 经验教训

> **生产环境永远不要直接操作，走审批流程是对自己负责**

## 🔥 案例 04：证书过期导致的全线崩溃

### 故障描述

```
时间：2023-01-01 00:00 (新年第一秒)
现象：所有 HTTPS 服务证书过期，无法访问
影响：新年第一天全线崩溃，持续 2 小时
```

### 根本原因

1. SSL 证书 1 年有效期，忘记续期
2. 没有证书过期告警
3. 证书分散在多台服务器，手动管理
4. 没有自动化证书轮换

### 时间线

```
00:00 证书过期，HTTPS 服务全部报错
00:05 监控检测到服务异常，但不知道是证书问题
00:10 用户投诉新年无法下单
00:15 运维接到电话，开始排查
00:20 发现是证书过期
00:30 手动更新证书，一台一台改
01:00 改了 20 台，还剩 50 台
01:30 写批量脚本推送证书
02:00 全部恢复
```

### 改进措施

| 问题 | 改进 | 负责人 | 状态 |
|------|------|--------|------|
| 无证书告警 | 证书过期前 30 天告警 | 张三 | ✅ |
| 手动管理 | 使用 cert-manager 自动管理 | 李四 | ✅ |
| 证书分散 | 统一使用 Vault 管理 | 王五 | ✅ |

### 经验教训

> **证书管理要自动化，别指望人能记住每个过期时间**

## 🔥 案例 05：负载均衡配置错误导致服务雪崩

### 故障描述

```
时间：2022-08-20 10:00
现象：修改 SLB 配置后，流量全部打到一台服务器
影响：服务器被打爆，雪崩效应，持续 30 分钟
```

### 根本原因

1. SLB 健康检查配置错误，误判其他节点不健康
2. 没有变更审批流程
3. 没有灰度验证
4. 监控没发现流量倾斜

### 时间线

```
10:00 运维修改 SLB 健康检查配置
10:02 配置生效，SLB 认为只有 1 台健康
10:05 流量全部打到这台，CPU 100%
10:08 服务超时，用户开始投诉
10:10 监控告警 CPU 高，但没发现是流量倾斜
10:15 发现 SLB 配置错误，立即回滚
10:20 流量恢复均衡，服务恢复
10:30 编写故障报告
```

### 改进措施

| 问题 | 改进 | 负责人 | 状态 |
|------|------|--------|------|
| 无变更审批 | SLB 配置变更需审批 | 张三 | ✅ |
| 无灰度验证 | 变更后先观察监控 | 李四 | ✅ |
| 流量监控缺失 | 增加节点流量分布监控 | 王五 | ✅ |

### 经验教训

> **负载均衡配置变更影响巨大，必须审批+灰度**
