## 🔐 安全最佳实践

### SSH 安全配置

```
# /etc/ssh/sshd_config

# 禁止 root 登录
PermitRootLogin no

# 禁止密码登录，只允许密钥
PasswordAuthentication no
PubkeyAuthentication yes

# 修改默认端口 (减少扫描)
Port 2222

# 限制登录用户
AllowUsers ops admin

# 登录超时
LoginGraceTime 30
MaxAuthTries 3

# 会话超时
ClientAliveInterval 300
ClientAliveCountMax 2
```

### 防火墙规则

```bash
# iptables 最小规则

# 允许 SSH
iptables -A INPUT -p tcp --dport 2222 -j ACCEPT

# 允许监控端口
iptables -A INPUT -p tcp --dport 9100 -s 10.0.100.0/24 -j ACCEPT

# 允许应用端口
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# 允许已建立的连接
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# 允许本地回环
iptables -A INPUT -i lo -j ACCEPT

# 默认拒绝
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT
```

## 📊 监控最佳实践

### 告警分级标准

```
P0 - 电话 + 短信 + IM
├── 全站不可用
├── 数据丢失
├── 安全入侵
└── 响应时间 < 5 分钟

P1 - 短信 + IM
├── 核心功能不可用
├── 性能严重下降 (>50%)
├── 数据库主从断开
└── 响应时间 < 15 分钟

P2 - IM
├── 非核心功能异常
├── 性能轻微下降 (>20%)
├── 单节点故障
└── 响应时间 < 1 小时

P3 - 邮件
├── 资源使用率高
├── 证书即将过期
├── 备份任务失败
└── 响应时间 < 24 小时
```

### 告警规则模板

```yaml
# 告警规则命名规范
<服务>-<指标>-<阈值>-<时间>

示例：
├── node-cpu-80p-5m      # CPU > 80% 持续 5 分钟
├── node-mem-85p-5m      # 内存 > 85% 持续 5 分钟
├── node-disk-15p-10m    # 磁盘 < 15% 持续 10 分钟
├── app-latency-1s-5m    # 延迟 > 1s 持续 5 分钟
├── app-error-5p-5m      # 错误率 > 5% 持续 5 分钟
├── db-replication-60s   # 复制延迟 > 60s
└── cert-expire-30d      # 证书 30 天内过期
```

## 🔄 发布最佳实践

### 发布时间窗口

```
推荐发布时间：
├── 周二、周三、周四 (避开周一周五)
├── 上午 10:00-12:00 (避开早上和下班)
├── 非高峰时段 (避开用户活跃时间)
├── 有人值班时段 (别在休息时间发)

禁止发布时间：
├── 周一 (周一问题多)
├── 周五 (周五发布周末加班)
├── 节假日前一天
├── 晚上 20:00 后
└── 无值班时段
```

## 💾 备份最佳实践

### 备份策略矩阵

| 数据类型 | 备份频率 | 保留周期 | 存储位置 |
|----------|----------|----------|----------|
| 数据库 | 每天 + binlog 实时 | 30 天 + 归档 | 本地 + OSS + 异地 |
| 应用配置 | 每次变更 | 永久 | Git + OSS |
| 用户文件 | 每天 | 90 天 | OSS + 异地 |
| 系统配置 | 每周 | 90 天 | Git |
| 日志 | 不备份 | 30 天滚动 | 本地 |

### 备份验证流程

```bash
#!/bin/bash
# backup-verify.sh - 备份验证

BACKUP_FILE=$1

echo "=== 备份验证 ==="

# 1. 文件完整性验证
gzip -t $BACKUP_FILE || exit 1
echo "✅ 文件完整性验证通过"

# 2. 恢复到临时库
mysql -u root -p"xxx" -e "DROP DATABASE IF EXISTS verify_test;"
gunzip -c $BACKUP_FILE | mysql -u root -p"xxx" -e "CREATE DATABASE verify_test;"
gunzip -c $BACKUP_FILE | mysql -u root -p"xxx" verify_test

# 3. 数据完整性验证
TABLE_COUNT=$(mysql -u root -p"xxx" verify_test -e "SHOW TABLES;" | wc -l)
echo "表数量：$TABLE_COUNT"

# 4. 清理临时库
mysql -u root -p"xxx" -e "DROP DATABASE verify_test;"
echo "✅ 临时库清理完成"

echo "=== 验证通过 ==="
```

## 🔧 配置管理最佳实践

### 配置分层

```
┌─────────────────────────────────────────────────────────────┐
│                    配置层次                                  │
│                                                             │
│  第一层：环境配置 (Environment)                             │
│  ├── prod / staging / dev                                   │
│  ├── 不同环境不同值                                         │
│  └── 环境变量传递                                           │
│                                                             │
│  第二层：应用配置 (Application)                             │
│  ├── 服务级别配置                                           │
│  ├── 数据库连接池                                           │
│  └── 缓存配置                                               │
│                                                             │
│  第三层：基础设施配置 (Infrastructure)                      │
│  ├── K8s Deployment                                         │
│  ├── Service 配置                                           │
│  └── Ingress 配置                                           │
│                                                             │
│  第四层：密钥配置 (Secrets)                                 │
│  ├── 数据库密码                                             │
│  ├── API 密钥                                               │
│  └── 证书                                                   │
└─────────────────────────────────────────────────────────────┘
```

### 配置变更流程

```
配置变更 → 审批 → 变更记录 → 灰度验证 → 全量生效

禁止：
├── 直接修改配置文件
├── 修改后不通知
├── 没有版本记录
└── 没有回滚方案
```

├── 架构文档 (Architecture)
│   ├── 系统架构图
│   ├── 网络拓扑图
│   └── 数据流图
│
├── 部署文档 (Deployment)
│   ├── 环境准备
│   ├── 安装步骤
│   └── 配置说明
│
├── 运维手册 (Operations)
│   ├── 日常巡检
│   ├── 常见问题

│
├── SOP 流程 (SOP)
│   ├── 发布流程

│   └── 演练流程

## 💡 我的一句话最佳实践

1. **监控先行** - 没监控的系统不要上线
2. **备份必验** - 不能恢复的备份等于没有
3. **发布必灰** - 不灰度的发布是赌博
4. **变更必审** - 审批是对自己负责
5. **文档必活** - 死文档不如没文档
6. **复盘必改** - 不改的复盘是形式主义
7. **容量必规** - 等满了就晚了
8. **安全必左** - 安全前置到设计
9. **成本必看** - 云账单每月 Review
10. **团队必学** - 技术更新太快
