## 📐 我的运维三原则

### 原则一：能自动的绝不手动

**为什么？** 人会累、会忘、会错，机器不会。

**怎么做？**
```
手动操作 1 次 → 写脚本
脚本跑 3 次 → 做成工具
工具用 10 次 → 集成平台
平台跑 100 次 → 加自愈合
```

**我踩过的坑：**
- 凌晨 3 点被叫起来重启服务 → 现在自动重启 + 自愈
- 手工改配置漏了一台 → 现在配置即代码
- 备份忘了一个库 → 现在备份自动验证

---

### 原则二：看不见的就管不了

**为什么？** 故障从来不会挑你方便的时候来。

**怎么做？**
```
监控分层：
├── 基础设施层 (CPU/内存/磁盘/网络)
├── 平台服务层 (K8s/DB/MQ/Cache)
├── 应用服务层 (API/延迟/错误率)
└── 业务指标层 (订单/支付/用户)

告警分级：
├── P0 - 电话+ 短信+IM (5 分钟响应)
├── P1 - 短信+IM (15 分钟响应)
├── P2 - IM (1 小时响应)
└── P3 - 邮件 (24 小时响应)
```

**我踩过的坑：**
- 磁盘满了才知道 → 现在 80% 就告警
- 用户投诉才知道挂了 → 现在 1 分钟发现
- 查问题翻半天日志 → 现在链路追踪秒定位

---

### 原则三：变更必出事，只是时间问题

**为什么？** 80% 的故障来自变更。

**怎么做？**
```
变更流程：
申请 → 评审 → 审批 → 灰度 → 验证 → 全量 → 观察

灰度策略：
1% → 10% → 50% → 100%
每步观察 10 分钟，有问题自动回滚

回滚机制：
一键回滚 < 5 分钟
数据回滚 < 30 分钟
```

**我踩过的坑：**
- 周五下午发布，周末加班修 → 现在禁止周五发布
- 没回滚方案，硬着头皮修 → 现在无回滚不发布
- 一个人发布，错了没人知道 → 现在双人复核

## 🔧 我的运维工具链

### 核心工具选择标准

| 标准 | 说明 | 我的选择 |
|------|------|----------|
| **开源优先** | 不被厂商绑定 | Prometheus/Grafana |
| **社区活跃** | 有问题能找到人 | Kubernetes |
| **简单易用** | 团队能快速上手 | GitLab CI |
| **可扩展** | 能跟着业务成长 | Terraform |

### ClawsOps 标准栈

```
┌─────────────────────────────────────────────────────────────┐
│  层次          工具选型                    替代方案         │
├─────────────────────────────────────────────────────────────┤
│  容器编排      Kubernetes                  Docker Swarm     │
│  CI/CD         GitLab CI + ArgoCD          Jenkins          │
│  监控          Prometheus + Grafana        Zabbix           │
│  日志          Loki + Grafana              ELK              │
│  链路          Jaeger                      Zipkin           │
│  配置          Nacos                       Apollo           │
│  密钥          Vault                       KMS              │
│  网络          Calico                      Flannel          │
│  网关          Nginx Ingress               Traefik          │
│  服务网格      Istio                       Linkerd          │
└─────────────────────────────────────────────────────────────┘
```

**为什么这么选？**
- Loki 比 ELK 轻量 10 倍，运维成本低
- GitLab CI 一体化，不用维护 Jenkins
- Prometheus 云原生标准，生态好

## 📋 我的运维 SOP

### 日常巡检 (每天 30 分钟)

```bash
#!/bin/bash
# daily-check.sh - 每日巡检脚本

echo "=== ClawsOps 每日巡检 ==="
echo "日期：$(date)"
echo ""

# 1. 核心服务状态
echo "【1】核心服务状态"
kubectl get pods -n production | grep -v Running | head -10

# 2. 资源使用 Top5
echo ""
echo "【2】CPU 使用 Top5"
kubectl top pods --all-namespaces --sort-by=cpu | head -5

echo ""
echo "【3】内存使用 Top5"
kubectl top pods --all-namespaces --sort-by=memory | head -5

# 3. 磁盘使用
echo ""
echo "【4】磁盘使用"
df -h | grep -E "^[a-z]|Filesystem"

# 4. 备份状态
echo ""
echo "【5】昨日备份状态"
ls -lt /backup/$(date -d yesterday +%Y%m%d)* | head -5

# 5. 告警汇总
echo ""
echo "【6】未恢复告警"
curl -s http://alertmanager:9093/api/v1/alerts | jq '.data[] | select(.status.state=="firing")'

echo ""
echo "=== 巡检完成 ==="
```

### 发布流程 (标准作业程序)

```markdown
# 发布检查清单

## 发布前
- [ ] 代码 Review 完成
- [ ] 测试通过 (单元/集成/压力)
- [ ] 回滚方案准备
- [ ] 通知相关人员
- [ ] 备份数据库

## 发布中
- [ ] 灰度 1% → 观察 10 分钟
- [ ] 灰度 10% → 观察 10 分钟
- [ ] 灰度 50% → 观察 10 分钟
- [ ] 全量发布 → 观察 30 分钟

## 发布后
- [ ] 核心功能验证
- [ ] 监控指标正常
- [ ] 日志无异常
- [ ] 用户反馈收集
- [ ] 发布报告编写
```

### 故障响应 (On-Call 流程)

```markdown
# 故障响应 SOP

## P0 故障 (5 分钟响应)
1. 接到告警 → 1 分钟内确认
2. 拉应急群 → 2 分钟内
3. 初步定位 → 5 分钟内
4. 临时恢复 → 15 分钟内
5. 根因修复 → 1 小时内
6. 故障复盘 → 24 小时内

## 故障复盘模板
1. 故障时间线 (精确到分钟)
2. 影响范围 (用户/业务/数据)
3. 根本原因 (5Why 分析)
4. 改进措施 (负责人 + 截止日)
5. 经验教训 (团队分享)
```

## 🚨 我的故障管理哲学

### 故障分级 (我定的)

| 级别 | 名称 | 响应 | 升级 | 示例 |
|------|------|------|------|------|
| **P0** | 灾难 | 5 分钟 | 立即 | 全站不可用 |
| **P1** | 严重 | 15 分钟 | 30 分钟 | 核心功能不可用 |
| **P2** | 一般 | 1 小时 | 2 小时 | 非核心功能异常 |
| **P3** | 轻微 | 4 小时 | 24 小时 | 性能下降/偶发错误 |
| **P4** | 建议 | 24 小时 | 1 周 | 优化建议/小问题 |

### 故障处理原则

```
1. 先恢复，后定位
   - 用户能用比什么都重要
   - 回滚不丢人，硬撑才丢人

2. 信息同步比技术重要
   - 每 15 分钟同步一次进展
   - 让老板知道你在干活

3. 复盘比处罚重要
   - 对事不对人
   - 改进措施比追责有用
```

## 💡 我的运维心得

### 10 条经验教训

1. **永远不要相信人工操作** - 能自动化的绝不手动
2. **监控要走在故障前面** - 等告警就晚了
3. **备份要定期恢复验证** - 不能恢复的备份等于没有
4. **变更要有回滚方案** - 没有回滚的发布是耍流氓
5. **文档要边做边写** - 事后补的文档没人看
6. **故障要彻底复盘** - 同样的坑不能踩两次
7. **容量要提前规划** - 等满了就晚了
8. **安全要左移** - 上线后再补安全漏洞成本太高
9. **成本要持续优化** - 云资源不用就关掉
10. **团队要持续学习** - 技术更新太快，不学就淘汰
